# Transformer Attention Accelerator - 機制與硬體規格 (v4)

**Target**: Xilinx KV260 FPGA (117K LUT, 1248 DSP, 4GB DDR4, 640 KB BRAM)
**Scope**: LLaMA-family decoder-only Transformer (TinyLlama, Qwen, LLaMA2/3, Mistral, ...)

---

## 1. 機制 (model-agnostic)

### Notation

| Symbol | 定義 |
|--------|------|
| `L` | 層數 (model-dependent) |
| `H_Q` | Query head 數 |
| `H_KV` | KV head 數 (GQA) |
| `D` | head_dim |
| `P` | num_pairs = D / 2 |
| `S` | sequence length |
| `R` | GQA ratio = H_Q / H_KV |
| `G` | KIVI V group size |
| `A` | active pairs per query |

### Layer-wise dispatch

| Layer | 處理機制 |
|-------|---------|
| Layer 0 | 異質管線, 跑滿 D dim, 8-bit Q·K + 8-bit V (不套 RAD) |
| Layer 1 ~ L-1 | RAD-Cascade (Q·K) + KIVI-V (V) |

Layer 0 為什麼異質: attention 分散度高 (per-row max ≈ 1/S), RAD 機制不適用. 此 layer 跑滿維度後續層才省 MAC.

### RAD-Cascade Stage 1: 主動 + 補償 (Q·K 端)

**Active path** (8-bit MAC):
- `A` active pairs per query (calibrated, default `A = round(0.375 × P)`)
- 由兩類 pair 組成:
  - `n_K` 個 K-anchor pair (per-token runtime running max of |K|)
  - `A - n_K` 個 dynamic pair (per-query top-K by |Q|)
- **Q-anchor 廢除** (跨 model 驗證冗餘, 見 2.5)
- 約束: `n_K ∈ {1, 2, 3, 4}` (4-bit register flexibility, 涵蓋 head_dim 64-128)

**SIGN compensation path** (1-bit XNOR + 8-bit scale):
- 對其他 `P - A` 個 dropped pair (`D - 2A` dim) 算:
- `score_comp = β × mean(|K_drop|) × Σ_d (|Q_d| × sign(Q_d) ⊕ sign(K_d))`
- β = 1.0 (Theorem 1 unbiased optimal)

**Theorem 1** (SIGN Compensation Unbiased):
取期望 E[sgn(K) × K | K symmetric] = mean(|K|), 故 β=1.0 為 unbiased estimator.

### RAD-Cascade Stage 2: Block dispatch

對每個 query, 把 K-side 切成 `B` block (`B = 16` default)

**Per block**:
- `block_max`, `block_mean`
- `signal_amp = block_max - block_mean`
- `margin = top1 - top2` of block scores

**Dispatch rule**:
- `bypass`: `margin > signal_amp × T_margin` → 只送 top-1 candidate
- `overflow`: `score ≥ block_max - signal_amp × T_overflow` → 候選, 最多 `M_local` 個 / block

**Default thresholds** (Chebyshev confidence interval derived):
- `T_margin = 0.75` (3σ confidence, P(error) ≤ 11%)
- `T_overflow = 0.25` (1σ tolerance, cover 70% candidates)
- `M_local = 3`

**Candidate 補算**: 對 candidate token 補算 dropped `D - 2A` dim 用 8-bit MAC.

### Sparsity Strategy 對比 SparQ

我們跟 SparQ (ICML 2024) 是**互補的 sparsity trade-off**:

| | SparQ | RAD-Cascade |
|--|------|------------|
| Dim selection | top-r \|q\| (~25-50% dim) | active 37.5% dim |
| Token recompute | top-k full KV (k=128, ~50%) | Block dispatch ~18% |
| Dim-side anchor | 無 | K-anchor 必要 |
| V output | mean V compensation | KIVI quantization |
| Hardware target | GPU / IPU (irregular fetch OK) | FPGA (regular BRAM access) |

**核心差異**: SparQ token-side 補算激進 (~50%), 不需 dim-anchor. 我們 token-side 補算保守 (~18%, FPGA 規則化), 需 K-anchor 補. 兩個架構都 work, 對應不同 hardware constraint.

實驗驗證 (TinyLlama HellaSwag):
- 0Q+0K (沒 anchor, 跟 SparQ Step 1 概念對齊): drop **4.37 pp**
- 0Q+1K (我們設計): drop **0.98 pp**

→ K-anchor 對我們機制 critical (~3.4 pp 真實貢獻).

### KIVI-V (V 端 quantization)

對每個 V token 做 per-token group-wise INT4 quantization (Liu et al., KIVI 2024).

**演算法**:
```
For each (B, H, S) token:
    For each group g of G consecutive V channels:
        v_min = min(V_token[g*G : (g+1)*G])
        v_max = max(V_token[g*G : (g+1)*G])
        scale_g = (v_max - v_min) / 15
        V_q[g] = round((V[g] - v_min) / scale_g)  ∈ [0, 15]   (INT4)
    Store (V_q, scale_g, v_min_g) per group
```

**Default**: `G = 32` (KIVI paper default)

### 完整 dataflow

```
Per new K_i (streaming during attention compute):
    Update K_max_running[L, kv_h, pair]  // 1 cycle max compare

Per query token:
    1. K-anchor: top-n_K of K_max_running
    2. Dynamic: per-query top-(A - n_K) of |Q|
    3. Q·K Stage 1 (active + SIGN)
    4. Q·K Stage 2 (block dispatch): 候選補算
    5. Softmax → attn_prob
    6. Group-wise dequantize V_int4 with scale_g
    7. attn_prob @ V_dequant → output
```

### Component contribution (TinyLlama, runtime_max + 0Q+1K base)

實驗 ablation 數據 (HellaSwag drop pp):
- Full mechanism: 0.97 pp (58.61%)
- No SIGN: 2.38 pp (drop SIGN +1.41 pp)
- No Block dispatch: 5.17 pp (drop Block +4.20 pp)
- Minimal (no SIGN, no Block): 23.12 pp (combined +22.15 pp)

**Block dispatch 是 dominant component (+4.20 pp), SIGN 是 supporting (+1.41 pp)**. Super-additive synergy (22.15 vs sum 5.61), 機制 component 必須共存.

## 2. 軟體超參如何測得 (model-agnostic methodology)

### 2.1 Metric Hierarchy (機制決策指標規範)

| Metric | 用途 | 可信度 |
|--------|------|-------|
| Spearman ρ | **僅** active dim ratio (A/P) sweep, 量化選擇 | 對機制存廢預測失效 (anchor on/off 案例: ρ 差 0.003, accuracy 差 3.33 pp). **不可作為機制決策指標** |
| KL divergence (max) | (n_K) sweep, K-anchor 數量決策 | 跨 model 驗證 (TinyLlama, Qwen) 預測正確. Primary metric |
| KL divergence (mean) | Secondary | Qwen 上預測 ranking 錯, 不如 max 可信 |
| MSE (score-level) | Sanity backup | 跟 KL max ranking 接近, noise 略大 |
| End-to-end accuracy | Ground truth, final 決策 | **任何機制變動必須 end-to-end 驗證** |

**Cutoff for "identical" judgment** (KL max gap):
- < 0.01: identical (no end-to-end needed)
- 0.01-0.1: 邊界 (建議 end-to-end)
- > 0.1: 顯著 differ (end-to-end mandatory)

**注意**: KL 是對數量級, 跟 accuracy 非線性映射. 「KL gap N → accuracy gap M」沒有 universal mapping, 只能用於 ranking, 不能用於量化預測 accuracy.

### 2.2 β (SIGN compensation coefficient)

**值**: 1.0 (Theorem 1 unbiased)
**方法**: Theorem 推導 + 200 sample sweep (β ∈ {0.3, 0.5, 0.7, 1.0}) 驗證

### 2.3 KIVI group size `G`

**值**: 32 (KIVI paper default)
**方法**: 引用 KIVI paper, 不重 sweep

### 2.4 active_dim ratio `A/P`

**值**: 0.375 (37.5%) default

**方法**: 
- 跨 model 經驗 sweet spot (TinyLlama, Qwen 同 ratio work)
- 落在 SparQ (ICML 2024) 驗證範圍 (r/d_h ∈ {25%, 50%}) 內
- 對新 model 部署: 用 25% / 37.5% / 50% 三個 candidate 跑 KL max + end-to-end, 取 drop < 1.5 pp 最小 ratio

**注意**: 沒有單一事前指標可預測 ratio 變化對 accuracy 影響. ρ 對 ratio 失效, KL max 對 ratio 變化未驗證可信度. 必須 end-to-end.

### 2.5 Q-anchor 廢除

**值**: n_Q = 0 (universal)

**證據**:
- TinyLlama: 1Q+1K (58.61%) vs 0Q+1K (58.60%), **gap 0.01 pp**
- Qwen2.5-1.5B: KL identical (0.0000 gap)
- SparQ Attention (ICML 2024) cross-model: Llama 2/3, Mistral, Gemma, Pythia 都用 pure dynamic Q top-r 無 anchor

**物理解釋**: Q magnitude per-token 變化大但 dim 位置 stable, dynamic top-K 自然涵蓋 layer-level outlier dim, Q-anchor 冗餘.

**硬體**: register `reg_n_Q_anchor` 保留 (4-bit), 默認 0, 可作為 future fallback (不需要重新 deploy RTL).

### 2.6 K-anchor 數量 `n_K`

**值**: per-model calibrated, n_K ∈ {1, 2, 3, 4}

**已驗證**:
- TinyLlama-1.1B (D=64, P=32): n_K = **1** (drop 0.97 pp)
- Qwen2.5-1.5B (D=128, P=64): n_K = **3** (drop 0.76 pp)
- LLaMA-7B (D=128, P=64): 預估 3, 待 end-to-end 驗證

**經驗法則**: n_K / P ≈ 3-5%, head_dim 越大 n_K 越大.

**Calibration 演算法**:
```
For n_K in {1, 2, 3, 4}:
    Compute KL max on 4 rep layers × 8 heads × 5 long samples
Pick top-2 candidate
End-to-end full HellaSwag on top-2
Select best
```

**約束放寬**: 舊規範 `n_Q + n_K ∈ {1, 2, 4, 8}` 廢除. K-anchor routing 本來就是 dynamic (per-token 不同 index), 2^k 對齊沒硬體效益.

### 2.7 K-anchor (Runtime Streaming)

**值**: per-token running max of |K|

**方法**: Streaming hardware update, **no offline calibration set required**

**演算法**:
```
Init: K_max[L, kv_h, pair] = 0  for all (L, kv_h, pair)

Per new token K_i (during attention compute, before KV cache write):
    K_pair_i[L, kv_h, p] = max(|K_i[L, kv_h, 2p]|, |K_i[L, kv_h, 2p+1]|)
    K_max[L, kv_h, p] ← max(K_max[L, kv_h, p], K_pair_i[L, kv_h, p])

Per query token (anchor selection):
    K_anchor_index = top-n_K of K_max[L, kv_h]
```

**Why runtime over offline** (TinyLlama validation):

| Aspect | Offline | Runtime-max |
|--------|---------|-------------|
| HellaSwag (short ctx) | 58.54% | 58.61% (tie) |
| PG-19 (long ctx, 2048 token) | PPL 12.096 (+5.22%) | **PPL 11.961 (+4.04%)** |
| Calibration set required | ✓ | ✗ |
| Attention sink (Xiao et al. 2024, StreamingLLM) protection | partial | automatic |
| HW cost | BRAM table only | BRAM table + max-compare unit |

**Long context advantage**: runtime -1.18% PPL on PG-19. Running max 自然抓 attention sink (BOS / 前幾 token 累積大 K).

### 2.8 mean(|K_drop|) per (layer, head)

**方法**: Per-batch streaming compute (runtime, no separate calibration)

```
Per attention forward pass:
    For each (L, head_kv):
        mean_K_drop[L, h] = mean over (batch tokens) × (dropped pair indices) of |K|
```

Cache in BRAM 16-bit fixed-point.

### 2.9 block_size `B`

**值**: B = 16 (Chebyshev P(error) < 5%, dispatch rate ~30%)

### 2.10 T_margin, T_overflow (dispatch thresholds)

**值**: T_margin = 0.75 (3σ, P(error) ≤ 11%), T_overflow = 0.25 (1σ, cover 70% candidates)

## 3. Calibration / Deployment Flow

### 3.1 Pre-deployment (offline, one-time per model, ~30 min)

1. Load target model + tokenizer
2. **Active dim ratio**: 25% / 37.5% / 50% sweep
   - For each: 用 default n_K (round(0.04 × P), at least 1)
   - 跑 KL max + 500 random sample HellaSwag (sanity)
   - Top-2 candidate
3. **n_K sweep** (KL max): n_K ∈ {1, 2, 3, 4}
   - Top-2 candidate
4. **Final validation** (full HellaSwag 10042)
   - Top-2 (ratio, n_K) combination
   - Pick best
5. **Long context sanity** (PG-19 window=2048, 1 book ~10 min)
   - Confirm Δ PPL < 5%
6. **Write registers**

### 3.2 Runtime (per inference, no calibration set)

- K-anchor: running max during streaming attention
- mean(|K_drop|): per-batch streaming compute
- KIVI G: default 32

## 4. 硬體實作 (parameterized RTL)

### 4.1 RTL 參數化常數

```systemverilog
// Compile-time parameters (model-portable)
parameter D_MAX        = 128;   // Max head_dim
parameter P_MAX        = 64;    // Max num_pairs
parameter L_MAX        = 32;    // Max num layers
parameter HQ_MAX       = 32;
parameter HKV_MAX      = 32;
parameter B_OPTIONS    = {8, 16, 32};
parameter ACTIVE_OPTIONS = {0.25, 0.375, 0.50};
parameter LANE_COUNT   = 6;
parameter PE_PER_LANE  = 8;
parameter KIVI_GROUP_SIZE = 32;
```

### 4.2 Runtime 配置 register

```
+----------------------------+
| reg_head_dim          [7:0]
| reg_num_layers        [5:0]
| reg_num_q_heads       [5:0]
| reg_num_kv_heads      [5:0]
| reg_active_lane_count [3:0]   // 1 ~ 6
| reg_n_K_anchor        [3:0]   // 0 ~ 8 (default 1-3)
| reg_n_Q_anchor        [3:0]   // 0 (default), reserved for future fallback
| reg_block_size        [5:0]   // 8 / 16 / 32
| reg_BETA              [3:0]   // β shift amount (β=1.0 → shift 0)
| reg_T_margin          [7:0]
| reg_T_overflow        [7:0]
| reg_M_local           [2:0]
+----------------------------+
```

### 4.3 BRAM tables (runtime updated)

| Table | Size | Update mechanism |
|-------|------|------------------|
| K_max_running | `L × HKV × P_MAX × 8` bit ≈ 0.7 KB | Per-token max-compare |
| mean_K_drop | `L × HKV × 16` bit ≈ 0.3 KB | Per-batch streaming |

Q_anchor_idx table 廢除 (n_Q = 0 universal). 如 future fallback, runtime 可從 mean(|Q|) per-batch 算, 不需 BRAM table.

Total runtime BRAM: < 1 KB.

### 4.4 整體架構

```
KV260 FPGA
├── QK-PU
│   ├── PE Lane × LANE_COUNT (each PE_PER_LANE PE)
│   ├── Cross-bar dim selector
│   ├── Block-Sum-Indexing scheduler
│   ├── signal_amp / margin runtime compute
│   └── K_max running-max update unit
├── Softmax
│   ├── Base-2 softmax
│   └── Reciprocal LUT
├── V-PU (KIVI-V)
│   ├── 16×16 systolic array (4-bit MAC)
│   ├── Per-group scale-extraction multiplier
│   └── Per-group min offset adder
├── Memory
│   ├── K Buffer (BRAM, K-stationary, 8-bit)
│   ├── V Buffer (BRAM, V-stationary, 4-bit + INT8 scale + INT8 min)
│   ├── Q Buffer (BRAM, 8-bit)
│   ├── K_max running + mean(|K_drop|) tables (BRAM, ~1 KB)
│   └── DDR4 (weights + KV overflow)
```

### 4.5 K-max Running-Max Update Unit

**Function**: Each new K_i triggers max compare and update `K_max[L, kv_h, pair]`.

```
Per K_i input vector [D bits]:
    Step 1: Pair-wise abs-max
        K_pair_i[p] = max(|K_i[2p]|, |K_i[2p+1]|)   // 8-bit compare, P parallel
    Step 2: Running-max update
        K_max[p] ← max(K_max[p], K_pair_i[p])
    Write back to BRAM K_max_running table.
```

**Resource**:
- Comparators: P × 2 (8-bit), shared across layers via FSM
- ~0.5 K LUT
- 0 DSP
- Pipelined 1 cycle per new K vector

### 4.6 PE Lane 設計

每 lane: `PE_PER_LANE` PE
- Active path: PE_PER_LANE × 8-bit MAC + tree adder → (11 + log2(PE_PER_LANE))-bit partial sum
- SIGN path: PE_PER_LANE × XNOR-AND + |Q| accumulate
- Time-multiplexed: active 跟 SIGN 共用 PE

跨 lane reduction tree, target 200 MHz.

### 4.7 Cross-bar dim selector

`LANE_COUNT × PE_PER_LANE` cascading 4:1 mux, ~1K LUT, single-cycle dim selection per query.

### 4.8 Softmax 硬體化

Base-2 softmax + Reciprocal LUT (取代 div). 不需 P_THRES logic.

### 4.9 KIVI-V 硬體實作

Per-group MAC with scale-extraction trick:
- 16×16 systolic array, INT4 V × INT8 P
- Group 累加完成後乘 scale 1 次 (省 G-1 個 dequant multiplier)
- V_data 跟 V_scale 在不同 BRAM bank, 並行讀取

### 4.10 Reconfigurability range

| 參數 | 範圍 | 控制 |
|------|------|------|
| `head_dim` D | 64, 80, 96, 128 | cross-bar mux 寬度 (compile-time) |
| `num_layers` L | 1 ~ L_MAX | runtime register |
| `active_dim` A | 0.25 D ~ 0.50 D | active_lane_count |
| `n_K` | 1 ~ 4 | runtime register (default per-model) |
| `n_Q` | 0 (default), 1-2 (future fallback) | runtime register |
| `block_size` B | 8 / 16 / 32 | block FSM parameter |
| `β` | 0 / 0.5 / 1.0 (shift) | shift amount register |
| `T_margin`, `T_overflow` | 8-bit each | register |
| `KIVI_group_size` | 32 (固定) | compile-time |

### 4.11 Layer 0 異質管線 mapping

硬體 `LANE_COUNT × PE_PER_LANE` = 48 物理 PE.

Layer 0 跑 `ceil(D / 48)` iteration:
- D=64: 2 iteration (48 + 16, 4 lane idle in iter2)
- D=128: 3 iteration

Layer 0 throughput 慢 2-3x, 但 only 1/L of layers, 整體影響 < 10%.

### 4.12 SDH (Software-Defined Hardware) 範圍

No offline calibration set per model. 部署只需:
- Runtime register 配置 (active_lane, n_K, β, block_size)
- RTL 不變
- K-anchor + mean tables runtime 自動 update

### 4.13 資源估算 (KV260 target)

| 模組 | LUT | DSP | BRAM |
|------|-----|-----|------|
| QK-PU (LANE × PE) | ~5K | LANE×PE | 0 |
| Cross-bar dim selector | ~1K | 0 | 0 |
| K_max running-max unit | ~0.5K | 0 | 0 |
| Softmax (base-2 + recip LUT) | ~1.5K | 4 | 0 |
| V-PU (16×16 systolic, 4-bit) | ~3K | 96 | 0 |
| Per-group scale extract | ~0.7K | 16 | 0 |
| BS scheduler | ~1K | 0 | 0 |
| Control + FSM | ~3K | 0 | 0 |
| K Buffer (8-bit) | 0 | 0 | 256 KB |
| V data Buffer (4-bit) | 0 | 0 | 128 KB |
| V scale Buffer | 0 | 0 | 32 KB |
| Q Buffer (8-bit) | 0 | 0 | 32 KB |
| K_max running + mean tables | 0 | 0 | ~1 KB |
| **Total** | **~15.5K** | **164** | **~450 KB** |

KV260 limit: 117K LUT (13%) / 1248 DSP (13%) / 640 KB BRAM (70%)

**Bottleneck**: BRAM 70%, DDR4 bandwidth.

### 4.14 Memory wall 應對

| 技術 | 收益 | 與 RAD+KIVI 相容 |
|------|------|-----------------|
| W4A8 weight (AWQ/GPTQ) | DDR4 BW 省 50% | ✓ orthogonal |
| KV cache compression | 已內建 (KIVI INT4) | ✓ |
| Hardware batching | DSP util ↑ | ✗ edge 不適 |

## 5. Simulation Caveats

### 5.1 GPU simulation 範圍

當前 GPU simulation 涵蓋:
- ✓ 機制邏輯 (dim selection, SIGN, block dispatch, KIVI, runtime K-max)
- ✓ V 4-bit group-wise RTN quantization
- ✗ INT8 active MAC quantization (未模擬, GPU 跑 FP32)
- ✗ INT8 累加溢位 (未模擬)
- ✗ Softmax 8-bit fixed-point (未模擬)

### 5.2 MAC counter 定義

- `total_baseline_macs`: full FP MAC count
- `total_stage1_macs`: active dim × MAC count
- `total_stage2_macs`: candidate token 補算 MAC
- `total_v_rad_macs`: `total_valid × D × 0.5` (4-bit-equivalent)

### 5.3 待補實驗

1. INT8 active MAC fake quantization 驗證 (~30 min)
2. AWQ + RAD + KIVI + runtime_max 完整疊加 (~75 min)
3. Cross-task on KIVI + runtime_max (ARC-E/C, PIQA, WG, ~2 hr)
4. LLaMA-7B baseline + RAD + KIVI + runtime_max + (n_K sweep) (~10-12 hr)

## 7. Spec Discipline (規範補充)

### 7.1 機制決策流程

1. **L1 (sweep filter)**: ρ on 4 rep layers, 排除明顯爛 candidate
2. **L2 (decision)**: KL max on 5 long samples × 4 rep layers × 8 heads
3. **L3 (verification)**: random 500 sample end-to-end, top-2 candidate
4. **L4 (publication)**: full HellaSwag 10042 + long context PG-19

### 7.2 約束驗證

每個 spec 約束必須有物理硬體理由:
- "對齊 2^k" 需要說明是 (a) routing width, (b) addressing depth, (c) other
- 沒明確硬體成本依據的約束, 該放寬讓 calibration 找最佳解

### 7.3 Regression 規則

- 已驗證 work 過的數據 (`*_final.json`) 直接引用, 不重跑
- 新機制變動只跑新 config, 不重跑舊 baseline
- 機制變動 cell 必須對齊 work cell 不重寫, 只加 toggle

### 7.4 Sanity sample 規則

- 不用 dataset 前 N (HellaSwag 前 500 acc_norm 比 full 低 9 pp)
- 用 random 抽 500 或 stratified sample

### 7.5 Methodology 規則

- 機制變更必須 end-to-end 在 short + long context 兩個場景驗證
- Runtime / streaming 機制要在長 context 上展現 advantage
- ρ metric 失效 case: anchor on/off, component 數量變化
- KL max 可信 case: (n_K) sweep on 同 active dim 量級
- KL max 未驗證 case: active ratio 變化 (dim count 變化), 必須 end-to-end
