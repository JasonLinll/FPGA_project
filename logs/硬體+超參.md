# Transformer Attention Accelerator - 機制與硬體規格 (v3)

**Target**: Xilinx KV260 FPGA (117K LUT, 1248 DSP, 4GB DDR4, 640 KB BRAM)
**Scope**: LLaMA-family decoder-only Transformer (TinyLlama, Qwen, LLaMA2, ...)

---

## 1. 機制 (model-agnostic)

### Notation

| Symbol | 定義 |
|--------|------|
| `L` | 層數 (model-dependent) |
| `H_Q` | Query head 數 |
| `H_KV` | KV head 數 (GQA) |
| `D` | head_dim |
| `P` | num_pairs = D / 2 |
| `S` | sequence length |
| `R` | GQA ratio = H_Q / H_KV |
| `G` | KIVI V group size |

### Layer-wise dispatch

| Layer | 處理機制 |
|-------|---------|
| Layer 0 | 異質管線, 跑滿 D dim, 8-bit Q·K + 8-bit V (不套 RAD) |
| Layer 1 ~ L-1 | RAD-Cascade (Q·K) + KIVI-V (V) |

Layer 0 為什麼異質: attention 分散度高 (per-row max ≈ 1/S), RAD 機制不適用. 此 layer 跑滿維度後續層才省 MAC.

### RAD-Cascade Stage 1: 主動 + 補償 (Q·K 端)

**Active path** (8-bit MAC):
- `A` active pairs per query (calibrated, default `A = round(0.375 × P)`)
- 由三類 pair 組成:
  - `n_Q` 個 Q-anchor pair (per-(layer, head), layer-level mean |Q| based)
  - `n_K` 個 K-anchor pair (per-(layer, head), **runtime running-max** of |K|)
  - `A - n_Q - n_K` 個 dynamic pair (per-query top-K by |Q|)
- 約束: `n_Q + n_K ∈ {1, 2, 4, 8}` (hardware-aligned 2^n)

**SIGN compensation path** (1-bit XNOR + 8-bit scale):
- 對其他 `P - A` 個 dropped pair (`D - 2A` dim) 算:
- `score_comp = β × mean(|K_drop|) × Σ_d (|Q_d| × sign(Q_d) ⊕ sign(K_d))`
- β = 1.0 (Theorem 1 unbiased optimal, 已跨 model 驗證)

**Theorem 1** (SIGN Compensation Unbiased):
取期望 E[sgn(K) × K | K symmetric] = mean(|K|), 故 β=1.0 為 unbiased estimator.

### RAD-Cascade Stage 2: Block dispatch

對每個 query, 把 K-side 切成 `B` block (`B = 16` default)

**Per block**:
- `block_max`, `block_mean` 算出來
- `signal_amp = block_max - block_mean`
- `margin = top1 - top2` of block scores

**Dispatch rule**:
- `bypass`: `margin > signal_amp × T_margin` → 只送 top-1 candidate
- `overflow`: `score ≥ block_max - signal_amp × T_overflow` → 候選, 最多 `M_local` 個 / block

**Default thresholds** (Chebyshev confidence interval derived):
- `T_margin = 0.75` (3σ confidence, P(error) ≤ 11%)
- `T_overflow = 0.25` (1σ tolerance, cover 70% candidates)
- `M_local = 3`

**Candidate 補算**: 對 candidate token 補算 dropped `D - 2A` dim 用 8-bit MAC.

### KIVI-V (V 端 quantization)

對每個 V token 做 **per-token group-wise INT4 quantization** (Liu et al., KIVI 2024).

**演算法**:
For each (B, H, S) token:
For each group g of G consecutive V channels:
v_min = min(V_token[g*G : (g+1)G])
v_max = max(V_token[gG : (g+1)*G])
scale_g = (v_max - v_min) / 15
V_q[g] = round((V[g] - v_min) / scale_g)  ∈ [0, 15]   (INT4)
Store (V_q, scale_g, v_min_g) per group

**Default**: `G = 32` (KIVI paper default)

### 完整 dataflow
Per new K_i (streaming during attention compute):
Update K_anchor_running_max[L, kv_h, pair]  // 1 cycle max compare
Per query token:
1. Q-anchor: layer-level top-n_Q of mean(|Q|)
2. K-anchor: top-n_K of K_anchor_running_max (exclude Q-anchor indices)
3. Dynamic: per-query top-(A - n_Q - n_K) of |Q|
4. Q·K Stage 1 (active + SIGN): D/A ratio MAC saving
5. Q·K Stage 2 (block dispatch): 候選補算
6. Softmax → attn_prob
7. Group-wise dequantize V_int4 with scale_g
8. attn_prob @ V_dequant → output (with scale-extraction累加)

## 2. 軟體超參如何測得 (model-agnostic methodology)

### 2.1 β (SIGN compensation coefficient)

**值**: 1.0 (Theorem 1 unbiased)
**方法**: Theorem 推導 + 200 sample sweep (β ∈ {0.3, 0.5, 0.7, 1.0}) 驗證

### 2.2 KIVI group size `G`

**值**: 32 (KIVI paper default)
**方法**: 引用 KIVI paper, 不重 sweep
**Note**: G 太小 (≤ 16) metadata 開銷大, G 太大 (≥ 128) 量化誤差累積. G=32 是 KIVI 已驗證的 sweet spot.

### 2.3 active_dim ratio `A/P`

**方法**: Spearman ρ sweep on representative layers (排除 Layer 0)

**演算法**:
Select 4-5 representative layers (排除 L0)
For each ratio in {12%, 18%, 25%, 31%, 37%, 50%, 62%}:
For each layer L in selection:
Compute RAD score with this ratio
Compute Spearman ρ vs FP32 score
record min ρ across layers
Pick smallest ratio where min ρ ≥ 0.90

**重要**: ρ 只用於初始 sweep 範圍篩選 (`A/P` 量化選擇), 不能作為機制存廢 (anchor on/off) 或同類 component 比較 (offline vs runtime) 的決策標準. **任何機制變動必須 end-to-end 驗證**.

**Safety margin**: 取最小可行 ratio. 跨 model 觀察 37.5% 為 sweet spot.

### 2.4 (n_Q, n_K) anchor 組合

**方法**: 完整 sweep on hardware-aligned combos

**約束**: `n_Q + n_K ∈ {1, 2, 4, 8}` (硬體 routing 對齊)

**演算法**:
For each (n_Q, n_K) with n_Q + n_K ∈ {1, 2, 4, 8}:
Compute worst-case Spearman ρ across representative layers
Pick combo with highest worst-case ρ
Further validate top-2 candidates with end-to-end accuracy on random 500 sample

**Observation**: 不同 model best combo 不同 (e.g., TinyLlama 1Q+1K). **必須 per-model calibrate**.

**已驗證**: TinyLlama 1Q+1K with runtime_max + SIGN + Block + KIVI = 58.61% (FP32 baseline 59.58%, drop 0.97 pp).

### 2.5 K-anchor (Runtime Streaming)

**值**: per-token running max of |K|

**方法**: Streaming hardware update, **no offline calibration set required**

**演算法**:
Init: K_max[L, kv_h, pair] = 0  for all (L, kv_h, pair)
Per new token K_i (during attention compute, before KV cache write):
K_pair_i[L, kv_h, p] = max(|K_i[L, kv_h, 2p]|, |K_i[L, kv_h, 2p+1]|)
K_max[L, kv_h, p] ← max(K_max[L, kv_h, p], K_pair_i[L, kv_h, p])
Per query token (anchor selection):
cal_k = K_max[L, kv_h].clone()
cal_k[Q_anchor_index] = -∞       // exclude Q-anchor occupied pair
K_anchor_index = top-n_K of cal_k

**Why runtime over offline**:

| Aspect | Offline | Runtime-max |
|--------|---------|-------------|
| HellaSwag (short ctx) | 58.54% | 58.61% (tie, +0.07) |
| PG-19 (long ctx, 2048 token) | PPL 12.096 (+5.22%) | **PPL 11.961 (+4.04%)** |
| Calibration set required | ✓ | ✗ |
| Attention sink (Xiao et al. 2024, StreamingLLM) protection | partial | **automatic** |
| HW cost | BRAM table only | BRAM table + max-compare unit |

**Physical interpretation**: Running max 自然抓 attention sink (BOS / 前幾個 token 的累積大 K), 對 long context outlier channel adaptive precise.

**Update**: K-anchor index per query token recomputed from `K_max` (per-token 不同, 因 running max 隨 token 累積).

### 2.6 Q-anchor index per (layer, head)

**方法**: Layer-level mean(|Q_pair|) per batch (no separate calibration set)
Per attention forward pass:
For each (L, head_q):
global_q[L, h, p] = mean over current batch tokens of (|Q[L, h, p, 0]| + |Q[L, h, p, 1]|)
Q-anchor index = top-n_Q of global_q (exclude any pre-selected pair)

**Why no separate calibration**: Q magnitude pattern stable across batches (verified across TinyLlama / Qwen), per-batch mean 足夠.

### 2.7 mean(|K_drop|) per (layer, head)

**方法**: Per-batch expected value (runtime, no separate calibration)
Per attention forward pass:
For each (L, head_kv):
mean_K_drop[L, h] = mean over (batch tokens) × (dropped pair indices) of |K|

**Note**: 跟 K-anchor 一樣 streaming compute, 不需 offline calibration. Cache in BRAM 16-bit fixed-point.

### 2.8 block_size `B`

**方法**: Chebyshev statistical bound + dispatch overhead trade-off

**Default**: `B = 16` (Chebyshev P(error) < 5%, dispatch rate ~30%)

### 2.9 T_margin, T_overflow (dispatch thresholds)

**方法**: Chebyshev confidence interval
T_margin = 0.75   → 3σ, P(error) ≤ 11%
T_overflow = 0.25 → 1σ, cover 70% candidates

## 3. Calibration / Deployment Flow (跨 model deployment, ~10 min)

### 3.1 Pre-deployment (offline, one-time per model)

1. Load target model + tokenizer
2. **Active dim sweep**: Spearman ρ on 4-5 representative layers (excluding L0)
3. **Anchor combo sweep**: completing `n_Q + n_K ∈ {1,2,4,8}`, worst-case ρ standard + random 500 sample end-to-end verify
4. **β sweep** (200 sample): verify β=1.0 optimal (theorem confirm)
5. **Write registers**: `active_lane_count`, `n_Q`, `n_K`, `β`, `T_margin`, `T_overflow`, `block_size`

### 3.2 Runtime (per inference, no separate calibration set)

- K-anchor: running max during streaming attention compute (no calibration data)
- Q-anchor: per-batch mean(|Q|) computed on-the-fly
- mean(|K_drop|): per-batch streaming compute
- KIVI G: 用 default G=32, 不需 calibrate

**Note**: Offline K-anchor calibration set **不再需要**. Anchor 跟 mean 表都 runtime 計算, 無 train/eval distribution mismatch concern.

## 4. 硬體實作 (parameterized RTL)

### 4.1 RTL 參數化常數

### 4.3 BRAM tables (runtime updated, no offline calibration)

| Table | Size | Update mechanism |
|-------|------|------------------|
| K_max_running | `L × HKV × P_MAX × 8` bit ≈ 0.7 KB | Per-token max-compare update |
| Q_anchor_idx | `L × HQ × n_Q × ceil(log2(P_MAX))` bit ≈ 0.1 KB | Per-batch top-K from mean(|Q|) |
| mean_K_drop | `L × HKV × 16` bit ≈ 0.3 KB | Per-batch streaming compute |

Total runtime BRAM: < 2 KB (negligible vs 448 KB KV buffer)

### 4.4 整體架構
KV260 FPGA
├── QK-PU
│   ├── PE Lane × LANE_COUNT (each PE_PER_LANE PE)
│   ├── Cross-bar dim selector
│   ├── Block-Sum-Indexing scheduler
│   ├── signal_amp / margin runtime compute
│   └── K_max running-max update unit (new)
├── Softmax
│   ├── Base-2 softmax
│   └── Reciprocal LUT
├── V-PU (KIVI-V)
│   ├── 16×16 systolic array (4-bit MAC)
│   ├── Per-group scale-extraction multiplier
│   └── Per-group min offset adder
├── Memory
│   ├── K Buffer (BRAM, K-stationary, 8-bit)
│   ├── V Buffer (BRAM, V-stationary, 4-bit data + INT8 scale + INT8 min)
│   ├── Q Buffer (BRAM, 8-bit)
│   ├── K_max running register table (BRAM, ~0.7 KB)
│   ├── Q_anchor + mean(|K_drop|) tables (BRAM, ~0.4 KB)
│   └── DDR4 (weights + KV overflow)

### 4.5 K-max Running-Max Update Unit (NEW)

**Function**: Each new K_i triggers max compare and update `K_max[L, kv_h, pair]`.
Per K_i input vector [D bits]:
Step 1: Pair-wise abs-max
K_pair_i[p] = max(|K_i[2p]|, |K_i[2p+1]|)   // 8-bit compare, P times in parallel
Step 2: Running-max update
K_max[p] ← max(K_max[p], K_pair_i[p])       // 8-bit compare per pair
Write back to BRAM K_max_running table.

**Resource**:
- Comparators: P × 2 (8-bit), shared across layers via FSM
- ~0.5 K LUT
- 0 DSP (max compare 用不到 multiplier)
- Pipelined 1 cycle per new K vector

### 4.6 PE Lane 設計 (unchanged)

每 lane: `PE_PER_LANE` PE
- Active path: PE_PER_LANE × 8-bit MAC + tree adder
- SIGN path: PE_PER_LANE × XNOR-AND + |Q| accumulate
- Time-multiplexed

跨 lane reduction tree, target 200 MHz

### 4.7 Cross-bar dim selector

`LANE_COUNT × PE_PER_LANE` cascading 4:1 mux, ~1K LUT

### 4.8 Softmax 硬體化

Base-2 softmax + Reciprocal LUT, 不需 P_THRES 比較 logic

### 4.9 KIVI-V 硬體實作

Per-group MAC with scale-extraction trick (unchanged from v2)

### 4.10 Reconfigurability range

## 5. Simulation Caveats

### 5.1 GPU simulation 範圍

當前 GPU simulation 涵蓋:
- ✓ 機制邏輯 (dim selection, SIGN compensation, block dispatch, KIVI quantization, runtime K-max)
- ✓ V 4-bit group-wise RTN quantization
- ✗ INT8 active MAC quantization (未模擬, GPU 跑 FP16/FP32)
- ✗ INT8 累加溢位 (未模擬)
- ✗ Softmax 8-bit fixed-point (未模擬)

### 5.2 MAC counter 定義

- `total_baseline_macs`: full FP MAC count
- `total_stage1_macs`: active dim × MAC count
- `total_stage2_macs`: candidate token 補算 MAC
- `total_v_rad_macs`: `total_valid × D × 0.5` (4-bit-equivalent)

### 5.3 待補實驗

1. SIGN / Block component contribution ablation (1Q+1K runtime_max base, ~5 hr)
2. INT8 active MAC fake quantization 驗證 (~30 min)
3. AWQ + RAD + KIVI + runtime_max 完整疊加 (~75 min)
4. Cross-task on KIVI + runtime_max (ARC-Easy, ARC-Challenge, PIQA, Winogrande, ~2 hr)
5. Qwen2.5-1.5B + RAD + KIVI + runtime_max (~75 min)
6. LLaMA-7B baseline + RAD + KIVI + runtime_max (~5-6 hr)

### 6.3 機制改版歷史

- **v1**: RAD-Cascade + PxV (probability-thresholded V quantization)
  - 已棄用: ZipCache (NeurIPS 2024) 論證 prob threshold 為不準確 importance metric
  - HellaSwag drop 0.98 pp, V saving 45.79%

- **v2**: RAD-Cascade + KIVI-V (offline K-anchor calibration)
  - HellaSwag drop 1.04 pp (tie with PxV), V saving 47.73%
  - PG-19 PPL +5.22%
  - First-principle 強, 引用 KIVI 2024

- **v3 (current)**: RAD-Cascade + KIVI-V + Runtime K-anchor (running max)
  - HellaSwag drop 0.97 pp (58.61%), V saving 47.73%
  - PG-19 PPL +4.04% (improvement over v2 offline +5.22%, **runtime -1.18% PPL on long context**)
  - 取代 offline calibration: 自動 streaming, 不需 calibration set, attention sink (StreamingLLM 2024) 自動保護
  - 硬體 cost trivial: ~0.5 K LUT, 0 DSP, 0.7 KB BRAM
  - 破解離線校準批評: 無 train/eval distribution mismatch
